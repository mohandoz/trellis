## Summary

Phase 3 creates the fixture infrastructure that Phase 4's regression suite will exercise. The work is entirely within the Conjure codebase itself ‚Äî no external libraries are needed. Three distinct deliverables must be produced: (1) nine per-profile fixture directories under `tests/fixtures/<profile>/`, each auditing green; (2) one intentionally-broken fixture at `tests/fixtures/_broken/` with an EXPECT file; and (3) two new scripts: `tests/lib/sandbox.sh` (isolation helper) and `scripts/regen-fixtures.sh` (fixture regenerator).

All critical behaviors have been verified against the live codebase. `conjure init --profile <p>` followed by audit produces deterministic, offline output. The `audit-setup.sh` script is self-contained (no `$CONJURE_HOME`, no network, no git, no graphify) and runs correctly when invoked directly against any directory containing a valid harness. The existing `tests/run.sh:194` already demonstrates the `mktemp -d + trap EXIT + env override` isolation pattern that `sandbox.sh` will codify.

**Primary recommendation:** Implement in three waves ‚Äî Wave 1 creates the two new scripts (sandbox.sh and regen-fixtures.sh), Wave 2 generates and commits all 9 green fixtures, Wave 3 creates the `_broken/` fixture and adds fixture audit assertions to `tests/run.sh`.

### Pattern 1: sandbox.sh ‚Äî setup-only form

**What:** Sources into callers (tests/run.sh and Phase 4's loop); caller controls output capture and trap lifetime. Sets env vars, copies fixture, returns sandbox path via `SANDBOX_DIR`.

**When to use:** Any time audit-setup.sh must run against a fixture without leaking to developer's real HOME.

```bash
# tests/lib/sandbox.sh ‚Äî sourced library (not executable)
# Usage: source tests/lib/sandbox.sh; sandbox_setup <fixture_dir>
# Sets SANDBOX_DIR and exports HOME/XDG_CONFIG_HOME/CLAUDE_CONFIG_DIR/PATH.
# Callers must: trap 'rm -rf "$SANDBOX_DIR"' EXIT

sandbox_setup() {
  local fixture_dir="$1"
  SANDBOX_DIR="$(mktemp -d)"
  cp -r "$fixture_dir/." "$SANDBOX_DIR/"
  export HOME="$SANDBOX_DIR"
  export XDG_CONFIG_HOME="$SANDBOX_DIR"
  export CLAUDE_CONFIG_DIR="$SANDBOX_DIR"
  # Restrict PATH: keep system tools (/usr/bin, /bin), add CONJURE_HOME/cli
  export PATH="$CONJURE_HOME/cli:/usr/local/bin:/usr/bin:/bin"
}
```

### Pattern 2: regen-fixtures.sh ‚Äî idempotent generation

**What:** For each profile, creates a seed dir with the appropriate manifest stub and a GENERATED-header CLAUDE.md, runs `conjure init --profile <p>`, then copies output to `tests/fixtures/<p>/`.

**When to use:** Run manually when profile templates change; safe to re-run.

```bash
#!/usr/bin/env bash
# scripts/regen-fixtures.sh ‚Äî regenerate all (or one) committed test fixtures.
# Usage: bash scripts/regen-fixtures.sh [--profile <p>]
set -euo pipefail
CONJURE_HOME="$(cd "$(dirname "$0")/.." && pwd)"
FIXTURES_DIR="$CONJURE_HOME/tests/fixtures"

regen_profile() {
  local p="$1"
  echo "[regen] $p"
  local seed; seed="$(mktemp -d)"
  trap 'rm -rf "$seed"' RETURN
  # Write manifest stub (profile-specific ‚Äî see manifest map)
  _write_manifest "$p" "$seed"
  # Write seed CLAUDE.md with GENERATED header + minimal content
  _write_seed_claude "$p" "$seed"
  # Run init + profile
  CONJURE_HOME="$CONJURE_HOME" "$CONJURE_HOME/cli/conjure" init --profile="$p" "$seed" >/dev/null
  # Copy to committed fixture location
  rm -rf "${FIXTURES_DIR:?}/$p"
  cp -r "$seed/." "$FIXTURES_DIR/$p/"
}
```

### Pattern 3: _broken/EXPECT ‚Äî grep pattern file

**What:** One extended-grep pattern per line. Phase 4's runner greps each pattern against the fixture's audit output. For Phase 3, a single pattern is sufficient.

**When to use:** Any fixture that should produce non-green audit output; green fixtures can have an EXPECT file asserting `PASS:` count or omit it entirely.

```
# tests/fixtures/_broken/EXPECT
HARD CAP exceeded
```

This matches the exact audit output: `‚úó CLAUDE.md: 201 lines (HARD CAP exceeded ‚Äî trim)` [VERIFIED: live run]

### Anti-Patterns to Avoid

- **Hand-crafting fixture `.claude/` by hand:** Every file in `.claude/` must be produced by `conjure init`. Hand-rolled fixtures would silently diverge from actual init output. Use `regen-fixtures.sh` exclusively.
- **Calling `cli/conjure audit` instead of `bash scripts/audit-setup.sh` in sandbox:** `conjure audit` calls `cmd_preflight` first, which invokes `scripts/preflight.sh` and uses `command -v` against PATH. This makes fixture runs non-deterministic when developer tools are present/absent. Call `audit-setup.sh` directly.
- **Setting `SANDBOX_DIR` to a path inside the committed fixture:** The committed fixture must never be mutated. Always copy to a fresh `mktemp -d`.
- **Using exit code alone to assert broken fixture:** `tests/run.sh` must grep for the specific finding string, not just check `[ $? -ne 0 ]`. This satisfies TEST-04's "assertions check specific findings" requirement.
- **Leaving `graphify-out/` in fixtures:** `audit-setup.sh` checks for `graphify-out/graph.json` and emits a warning if present. Fixtures must not include this directory to keep audit output deterministic.
- **Including a CLAUDE.md with @imports:** `audit-setup.sh` greps for `^@` lines and emits an error. Fixture CLAUDE.md files must have no `@imports`.

## Critical Findings (Verified Against Live Code)

### Finding 1: `conjure init` does NOT create CLAUDE.md

[VERIFIED: live run + `scripts/init-project.sh` comments]

`init-project.sh` explicitly states: "Does NOT write CLAUDE.md content." The `profiles/*/apply.sh` appends its fragment to CLAUDE.md only IF CLAUDE.md already exists (guarded by `[ -f "$TARGET/CLAUDE.md" ]`).

**Consequence for regen-fixtures.sh:** The script must create a seed CLAUDE.md before running `conjure init`. It cannot rely on init to create one. The seed CLAUDE.md should contain:
1. The `# GENERATED ‚Äî do not edit directly; run scripts/regen-fixtures.sh` header (D-12)
2. Minimal project content (~20 lines) so the profile fragment has something to append to
3. After init+profile: total is 41‚Äì45 lines per profile (verified for all 9) ‚Äî safely under the 100-line cap

### Finding 2: CLAUDE.md size thresholds (audit behavior)

[VERIFIED: live runs]

`audit-setup.sh` has three size bands:
- `‚â§100 lines` ‚Üí `ok` (green, exit 0)
- `101‚Äì200 lines` ‚Üí `warn "within hard cap but over practical limit"` (exit 1, non-zero)
- `>200 lines` ‚Üí `err "HARD CAP exceeded ‚Äî trim"` (exit 2, error)

The CONTEXT says "pad to 105+ lines" (D-08). This triggers a **WARN** (exit 1), not an ERR. For the broken fixture:
- **Use 201+ lines** to trigger the `err` path (exit 2). This produces a cleaner test: `FAIL: 1`, exit code 2, distinct from the warn path.
- The EXPECT pattern `HARD CAP exceeded` matches the exact `err` message.

### Finding 3: audit-setup.sh is fully isolated from environment

[VERIFIED: grep of audit-setup.sh for `$CONJURE_HOME`, `$HOME`, `$PATH`, `$XDG`]

`audit-setup.sh` uses no environment variables ‚Äî it operates only on the TARGET directory argument. It calls: `wc`, `grep`, `find`, `jq`, `date`, `stat`, `basename` ‚Äî all system stdlib. This means sandbox isolation of `HOME`/`XDG_CONFIG_HOME`/`CLAUDE_CONFIG_DIR` is "belt and suspenders" for correctness (the audit itself won't accidentally read `~/.claude/`), but the PATH restriction matters for preventing spurious tool detection in future audit extensions.

### Finding 4: Profiles with no preflight.sh (four profiles)

[VERIFIED: live filesystem check]

Four profiles have NO `preflight.sh`:
- `node-nest` ‚Äî apply.sh does NOT call preflight (no `"$PROFILE_DIR/preflight.sh"` line)
- `monorepo` ‚Äî same
- `polyglot` ‚Äî same
- `data-science` ‚Äî same

Five profiles DO call their own `preflight.sh`:
- `ts-next`, `java-spring`, `rust-axum`, `go-gin`, `python-fastapi`

**Consequence:** When `regen-fixtures.sh` runs `conjure init --profile=<p>` for the four no-preflight profiles, the init will NOT invoke any profile-level tool check. These fixtures are simpler to generate. For the five profiles with preflight, the global preflight (node, git) runs but profile-specific tool checks (pnpm, cargo, go, etc.) are in the profile's own preflight.sh ‚Äî these only emit warnings, they don't block init.

### Finding 5: java-spring installs a bash .sh hook (not a regression for Phase 3)

[VERIFIED: `profiles/java-spring/apply.sh` + `profiles/java-spring/hooks/post-edit-format.sh`]

`java-spring/apply.sh` copies `hooks/post-edit-format.sh` (bash) to `.claude/hooks/`. This is a legacy bash hook from before Phase 1's SAFE-03 fix. However, `audit-setup.sh` only validates `.mjs` hooks (it `find`s with `-name '*.mjs'`). The `.sh` file does not cause an audit failure.

**Consequence for java-spring fixture:** The fixture will contain `post-edit-format.sh` alongside the five `.mjs` hooks. Audit still exits 0. This is acceptable for Phase 3 ‚Äî fixing this `.sh` hook is out of scope.

### Finding 6: monorepo fixture requires a `packages/` subdirectory in seed

[VERIFIED: `profiles/monorepo/apply.sh` reads `DETECTED` dirs]

`monorepo/apply.sh` checks for `packages/`, `apps/`, `services/`, `libs/` in TARGET. If none are found, it exits 1 (warning). The regen script must create at least one package subdir (e.g., `packages/api/`) in the seed dir so the monorepo profile can scaffold the nested CLAUDE.md.

**Consequence for regen-fixtures.sh:** For the `monorepo` profile seed:
```bash
mkdir -p "$seed/packages/api"
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Fixture content | Custom harness files by hand | `conjure init --profile <p>` | Hand-rolled fixtures diverge from real init output; regen-fixtures.sh keeps them in sync |
| Temp dir isolation | Custom `mkdir /tmp/...` logic | `mktemp -d` | POSIX-standard, race-condition safe; already used in tests/run.sh |
| Fixture cleanup | Manual `rm -rf` calls | `trap 'rm -rf "$SANDBOX_DIR"' EXIT` | EXIT trap fires on error, return, and normal exit ‚Äî manual calls miss failure paths |
| Audit invocation | Parsing audit output with awk/sed | Direct `grep` against captured output | audit-setup.sh output is line-oriented; grep is sufficient and already the project's pattern |

**Key insight:** The goal is not to build test infrastructure ‚Äî it is to produce committed fixtures that can be handed to a simple grep+exit-code assertion loop. Less infrastructure is better.

## Common Pitfalls

### Pitfall 1: Empty CLAUDE.md causes fixture audit to fail

**What goes wrong:** If regen-fixtures.sh creates the seed dir without a CLAUDE.md, `conjure init` produces a fixture with no CLAUDE.md. `audit-setup.sh` then emits `‚úó CLAUDE.md missing` (exit 2). The fixture is never green.

**Why it happens:** `init-project.sh` explicitly does NOT create CLAUDE.md. Profile `apply.sh` is guarded by `[ -f "$TARGET/CLAUDE.md" ]` ‚Äî it silently skips the fragment append.

**How to avoid:** `regen-fixtures.sh` must write a seed CLAUDE.md before invoking `conjure init`. The seed must contain the GENERATED header plus minimal sections so the profile fragment has somewhere to append.

**Warning signs:** Fixture audit exits 2 with `CLAUDE.md missing` error.

### Pitfall 2: Using 105-line CLAUDE.md for _broken/ hits WARN not ERR

**What goes wrong:** CONTEXT D-08 says "pad CLAUDE.md to 105+ lines." At 105 lines, `audit-setup.sh` emits a WARN (exit 1), not an ERR (exit 2). If Phase 4's runner asserts `FAIL: 1`, it will fail because FAIL count is 0 (only WARN count is 1).

**Why it happens:** The audit has three bands: ‚â§100 (ok), 101‚Äì200 (warn), >200 (err). D-08's "105+" was under-specified.

**How to avoid:** Use 201+ lines for `_broken/CLAUDE.md` to trigger the ERR path. This produces `FAIL: 1`, exit 2, and the message `HARD CAP exceeded ‚Äî trim` that the EXPECT pattern can match.

**Warning signs:** Audit of `_broken/` exits with code 1 instead of 2; EXPECT pattern `HARD CAP exceeded` matches nothing.

### Pitfall 3: Calling `cli/conjure audit` instead of `bash audit-setup.sh` in sandbox

**What goes wrong:** `conjure audit` calls `cmd_preflight` (which invokes `scripts/preflight.sh`) before running audit. Preflight uses `command -v` against the restricted sandbox PATH. If PATH excludes `jq`, preflight emits a warning and the test output becomes non-deterministic across machines.

**Why it happens:** The sandwich of preflight + audit is the user-facing flow, not the test flow.

**How to avoid:** `sandbox.sh` should invoke `bash "$CONJURE_HOME/scripts/audit-setup.sh" "$SANDBOX_DIR"` directly, bypassing the CLI dispatch and preflight.

**Warning signs:** Test output varies by developer machine; tests fail on CI when optional tools (jq, shellcheck) are absent.

### Pitfall 4: monorepo profile exits 1 if no packages/ dir in seed

**What goes wrong:** `profiles/monorepo/apply.sh` checks for `packages/`, `apps/`, `services/`, `libs/` and exits 1 if none detected. This propagates through the `|| echo "  ‚öÝ preflight had warnings"` guard in apply.sh but the init still exits 0 ‚Äî HOWEVER, the monorepo fragment is NOT appended (the echo exits apply.sh after the warning). The fixture ends up without the monorepo fragment.

**Why it happens:** monorepo apply.sh is different from other profiles: it returns early on missing dirs.

**How to avoid:** The monorepo seed must include `mkdir -p packages/api` before running `conjure init --profile=monorepo`. The fixture then gets the nested `packages/api/CLAUDE.md` as part of its committed content.

**Warning signs:** `tests/fixtures/monorepo/CLAUDE.md` does not contain `<!-- profile:monorepo -->`.

### Pitfall 5: Sandbox CONJURE_HOME must stay real (not sandboxed)

**What goes wrong:** If `CONJURE_HOME` is exported to the sandbox temp dir, `audit-setup.sh` (which does NOT use `$CONJURE_HOME`) is unaffected ‚Äî but any future test that invokes the CLI within the sandbox will fail because the CLI looks up `$CONJURE_HOME/scripts/` etc.

**Why it happens:** `sandbox.sh` blanket-exports `HOME`/`XDG_CONFIG_HOME`/`CLAUDE_CONFIG_DIR` ‚Äî it's tempting to also set `CONJURE_HOME`.

**How to avoid:** `sandbox.sh` must NOT override `CONJURE_HOME`. The kit location is real; only the fixture target HOME is sandboxed.

**Warning signs:** Tests that call `cli/conjure` inside the sandbox fail with "No such file" errors on kit scripts.

## Code Examples

### sandbox_setup invocation pattern (in tests/run.sh)

```bash
# Source the library at top of tests/run.sh
source "$CONJURE_HOME/tests/lib/sandbox.sh"

# In the fixture audit section:
echo "‚ñ∏ Fixture audits (sandboxed)"
for fx in "$CONJURE_HOME/tests/fixtures"/[^_]*/; do  # [^_] skips _broken/
  prof=$(basename "$fx")
  sandbox_setup "$fx"
  trap 'rm -rf "$SANDBOX_DIR"' EXIT
  AUDIT_OUT="$(bash "$CONJURE_HOME/scripts/audit-setup.sh" "$SANDBOX_DIR" 2>&1 || true)"
  AUDIT_RC=$?
  if [ "$AUDIT_RC" -eq 0 ]; then
    pass "fixture audit green: $prof"
  else
    fail "fixture audit non-green (rc=$AUDIT_RC): $prof"
    printf '%s\n' "$AUDIT_OUT" | head -5
  fi
done

# Broken fixture: assert specific finding
echo "‚ñ∏ Broken fixture (must fail audit with specific finding)"
sandbox_setup "$CONJURE_HOME/tests/fixtures/_broken"
trap 'rm -rf "$SANDBOX_DIR"' EXIT
BROKEN_OUT="$(bash "$CONJURE_HOME/scripts/audit-setup.sh" "$SANDBOX_DIR" 2>&1 || true)"
BROKEN_RC=$?
if [ "$BROKEN_RC" -ne 0 ]; then
  pass "_broken: audit exits non-zero (rc=$BROKEN_RC)"
else
  fail "_broken: audit should exit non-zero"
fi
while IFS= read -r pattern; do
  [ -z "$pattern" ] && continue
  [[ "$pattern" == \#* ]] && continue
  if printf '%s\n' "$BROKEN_OUT" | grep -qE "$pattern"; then
    pass "_broken: found expected finding: $pattern"
  else
    fail "_broken: missing expected finding: $pattern"
  fi
done < "$CONJURE_HOME/tests/fixtures/_broken/EXPECT"
```

## Open Questions

1. **Should regen-fixtures.sh verify audit-green before committing each fixture?**
   - What we know: nothing stops a broken profile template from producing a non-green fixture
   - What's unclear: should regen fail loudly if any fixture audit exits non-zero?
   - Recommendation: Yes ‚Äî add `bash "$CONJURE_HOME/scripts/audit-setup.sh" "$dest"` check after copy, exit with error if non-zero. This makes regen self-validating.

2. **Should committed fixtures include docs/ templates?**
   - What we know: `init-project.sh` creates `docs/ARCHITECTURE.md`, `docs/RUNBOOK.md`, `docs/adr/`, `.env.example`. `audit-setup.sh` checks for these and emits WARN if missing.
   - What's unclear: Are the generated `docs/` files noise in the fixture diffs?
   - Recommendation: Include them (they're output of `conjure init` and audit checks for them). Excluding them would require audit-setup.sh changes out of scope for this phase.

3. **EXPECT file format: should green fixtures have one too?**
   - What we know: CONTEXT D-09 says "green fixtures CAN have an EXPECT file too, asserting `0 errors` or similar." Phase 4's EXPECT comparison loop is the consumer.
   - What's unclear: Phase 3 should establish the format even if Phase 4 fully uses it.
   - Recommendation: Leave green fixtures without EXPECT files for now. Phase 4 can add them when the golden-file loop is built. Only `_broken/EXPECT` is required for Phase 3.
