<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| README.md → GitHub CDN | demo.gif is served from .github/assets/ by GitHub's CDN; no code execution risk |
| CI test job → committed file | test -s is a read-only assertion; no write surface in CI |
| committed binary → repository | demo.gif is a binary file; its size must be kept < 5 MB to avoid git performance issues |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | Tampering | demo.gif committed binary | accept | GIF is read-only CDN asset; no code execution path; magic bytes checked locally before commit via `file` command |
| T-05-06 | Denial of Service | demo.gif binary size bloat in git | mitigate | agg --idle-time-limit 2 + --speed 1.5 bounds GIF size; size printed by record-demo.sh for contributor review; target < 5 MB; checkpoint step requires human to verify size before committing |
| T-05-07 | Tampering | README.md img src points to committed asset | accept | asset is same-repo relative path (.github/assets/demo.gif); no external URL; GitHub CDN serves it; no SSRF surface |
| T-05-08 | Spoofing | CI could be tricked by an empty demo.gif | mitigate | `test -s` checks file is non-empty (size > 0); an attacker would need write access to the repository to replace the file — treated as out-of-scope for this threat model |
| T-05-SC | Tampering | npm/pip/cargo installs | accept | no package manager installs in this plan; only file edits and a committed binary |
</threat_model>
