      windows-hook-wiring:
        runs-on: windows-latest
        steps:
          - uses: actions/checkout@v4

          - name: Scaffold fixture
            shell: bash
            run: |
              mkdir -p /tmp/fixture
              CONJURE_HOME="$GITHUB_WORKSPACE" cli/conjure init /tmp/fixture

          - name: Assert node version
            shell: bash
            run: node --version

          - name: Assert node hook wiring in settings.json
            shell: bash
            run: grep 'node' /tmp/fixture/.claude/settings.json

          - name: Assert no bash hook regression
            shell: bash
            run: |
              if grep 'bash .claude/hooks' /tmp/fixture/.claude/settings.json; then
                echo "FAIL: bash hook commands found (SAFE-03 regression)" && exit 1
              fi

    Critical constraints per D-11 and RESEARCH.md:
    - Every step MUST have `shell: bash` — without it, GitHub Actions defaults to PowerShell
      on windows-latest, which breaks POSIX path handling
    - Do NOT add `apt-get`, `choco`, `npm install`, `actions/setup-node`, or any dep installation
      steps — per D-11 "No extra dependency installation"
    - Do NOT run `bash tests/run.sh` in this job — shellcheck is not pre-installed on windows-latest
      (RESEARCH.md Pitfall 4)
    - Use `CONJURE_HOME="$GITHUB_WORKSPACE"` (not `$PWD` or a hardcoded path) — matches the
      pattern used in the existing `audit-on-fixture` job at line 44
    - Use `/tmp/fixture` as the target directory — same convention as `audit-on-fixture` job
    - `grep 'node' /tmp/fixture/.claude/settings.json` must find the `node .claude/hooks/*.mjs`
      wiring; this assertion passes because Phase 1 fixed the template (SAFE-03)
    - The negative assertion uses `if grep ...; then ... && exit 1; fi` pattern — if bash hook
      commands ARE found, the job fails with a meaningful error message

    The `uses: actions/checkout@v4` step does not need `shell: bash` — it is a composite action
    with no shell context.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External PR → CI (windows-hook-wiring job) | An external contributor's PR triggers the windows-latest job, executing `cli/conjure init` against code from the PR branch. |
| cli/conjure init → /tmp/fixture | The init command writes files to /tmp/fixture on the Windows runner; the subsequent grep assertions read those files. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-W1 | Tampering | CI poisoning: external PR triggers windows-hook-wiring job | mitigate | The job runs only `cli/conjure init` (no secrets, no external network, no credentials). The worst a malicious PR can do is make the job fail. No secrets are exposed. Consider adding `permissions: contents: read` to the job if the project adds workflow-level permissions later. |
| T-04-W2 | Tampering | Script injection via /tmp/fixture path | accept | `/tmp/fixture` is a static hardcoded path — not interpolated from external input. The grep assertions read the file; they do not execute its contents. |
| T-04-W3 | Spoofing | windows-latest Node version changes | accept | `node --version` only asserts exit 0 (Node is present), not a specific version. A Node version bump does not break this assertion. |
| T-04-W4 | Information Disclosure | settings.json contents logged in CI | accept | settings.json contains no secrets — it is a template-generated file with hook wiring paths only. Logging it on CI failure is safe. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan per D-11. N/A. |
</threat_model>
