        sandbox_setup "$fx"

            pass "$prof EXPECT: $pattern"

            fail "$prof EXPECT: missing pattern: $pattern"
          fi
        done < "$expect_file"
      done

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| EXPECT file → grep -qE | EXPECT file content is read and each line used as a grep-E regex pattern. A malicious EXPECT file could contain patterns that always match (defeating test purpose) or patterns with catastrophic backtracking. |
| mktemp -d (fixture dirs) | Not applicable to this plan — this plan uses sandbox_setup (existing helper), not synthetic mktemp dirs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | tests/fixtures/*/EXPECT | accept | EXPECT files are committed to git — any tampering is visible in PR diff review. Patterns are trivially auditable (3 lines each). |
| T-04-02 | Tampering | tests/run.sh EXPECT loop: `grep -qE "$pattern"` | mitigate | Patterns come from committed files (not user input at runtime). The `case "$pattern" in \#*) continue ;; esac` guard prevents comment lines from being evaluated. The loop uses `printf '%s\n' "$AUDIT_OUT" | grep -qE` so the pattern is not shell-interpolated as a command. |
| T-04-03 | Denial of Service | grep -qE with regex catastrophic backtracking | accept | All three patterns (`PASS: [0-9]`, `WARN: 0`, `FAIL: 0`) are trivially simple — no nested quantifiers, no alternation on variable-length strings. Risk is negligible. |
| T-04-04 | Information Disclosure | EXPECT files containing absolute sandbox paths | mitigate | Per D-02, patterns must avoid absolute paths. Patterns are `PASS: [0-9]`, `WARN: 0`, `FAIL: 0` — semantic-only, no path disclosure. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan. N/A. |
</threat_model>
